# Production Recommendation

For a production deployment, the next step would be to strengthen durability, politeness, and observability. The current SQLite-based single-node design is a good fit for a localhost assignment, but a production crawler would benefit from a more durable queue or database-backed frontier, stronger retry policies, per-host throttling, robots.txt support, structured logging, and metrics for crawl throughput, failures, and index freshness. Resume support should also be made more robust by persisting enough worker and frontier state to recover safely after interruption.

If search must remain available while indexing at a larger scale, the incremental indexing model used here is still the right direction, but the storage and serving layers should be separated more clearly. A practical next step would be to keep the current architecture on a single machine while replacing the simple local index with a more scalable search-oriented backend, adding better ranking, duplicate-content detection, and more explicit operational tooling. The current design already separates crawling, storage, and search concerns, so it provides a good foundation for that evolution.
